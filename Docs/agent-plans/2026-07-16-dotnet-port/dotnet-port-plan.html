Overview

SwiftFITS is a FITS (Flexible Image Transport System) file parser and serializer written in Swift 6: roughly 3,680 lines across 11 source files and ~4,424 lines across 13 test files (231 @Test functions using the Swift Testing framework). The library reads FITS files into a model of sections (a primary header, optional extension headers, and their data segments), each header being a list of 80-byte properties (keyword / value / comment records), and serializes that model back to standards-compliant bytes with byte-for-byte round-tripping of unmodified content.

This plan ports the whole library and its tests to C# / .NET 10. The approach is dependency-ordered and incremental: each milestone ports one layer (or one concern of a large type), adds its tests, and leaves the solution compiling with all tests green, so every milestone can be reviewed and committed on its own. The port favors idiomatic C# over a mechanical translation while preserving the public API shape, type structure, names, and observable behavior wherever practical, so a SwiftFITS user recognizes the C# API. The scaffolding — solution (DotNetFITS.slnx), class library (DotNetFITS), and xUnit v3 test project (DotNetFITSTests) with InternalsVisibleTo, nullable reference types, and warnings-as-errors — is already in place.

Questions & answers

The design decisions settled with the maintainer before planning:

1. What C# type should represent binary data (Swift's Foundation Data)?
ReadOnlyMemory<byte>. It is the faithful analog of a Swift Data slice — a value type wrapping (buffer, offset, length) whose .Slice() is O(1) and shares storage, and which can be stored in a field. This preserves SwiftFITS's zero-copy behavior (chunking a large file into 2880-byte blocks, and holding data segments, without duplicating the bytes). Extension methods may be added on ReadOnlyMemory<byte> and/or byte[] to mirror the Swift standard library helpers where applicable and appropriate. ReadOnlySpan<byte> is used for local hot paths only (it is a stack-only ref struct and cannot be a field).
2. How should Swift's FITSError enum-with-associated-values map to C#?
A single FITSException : Exception carrying a FITSErrorKind discriminator and payload, with static factory methods per case (e.g. FITSException.DataError(reason), FITSException.InvalidBlockSize(size)). This keeps the one-error-type structure of the Swift enum and lets tests assert Assert.Throws<FITSException>, matching Swift's #expect(throws: FITSError.self). The maintainer's usual C# convention (“avoid throwing from public methods”) is deliberately overridden here because the source document mandates exception-based error handling “as in SwiftFITS,” and the entire API surface is throws.
3. What C# shape should FITSValue take?
A readonly struct with static factory methods, a FITSValueKind discriminator, and nullable typed accessors, implementing IEquatable<FITSValue> with the Swift equality quirk that NaN == NaN (and a matching GetHashCode). A value type is the closest match to Swift's value-type enum, is consistent with the ReadOnlyMemory<byte> choice, and avoids allocating a small heap object for every header value. The internal layout should reuse a single 8-byte numeric slot (reinterpreting bits for bool/long/double) plus one string reference, rather than one field per kind, to keep the struct lean.

Notes

Conventions

  • Naming: normal .NET conventions (PascalCase members, I-prefixed interfaces), but keep SwiftFITS formatting conventions (brace and alignment style) as the source document requests. One public type per file, organized into subfolders.
  • Visibility: library types are explicitly public by default (per the maintainer's convention); reserve internal/private for genuine implementation details. Members mutate through get/set properties rather than fields.
  • Doc comments: use C# XML documentation comments — /// followed by XML tags (<summary>, <param>, <returns>, <remarks>, <see cref>) — on every type and member, including internal/private ones. The /// delimiter is the standard C# convention (per the Microsoft C# reference and the default project templates), not a SwiftDoc artifact: Swift and C# share the /// marker but differ in body syntax — C# uses XML tags where SwiftDoc/DocC uses Markdown. (Recorded as a reference note for future ports.)
  • License header: every file carries the MIT header block from SwiftFITS (“Copyright (c) 2026, Jean-David Gadina - www.xs-labs.com”).
  • Culture independence: all FITS parsing and formatting must be deterministic and culture-invariant — use CultureInfo.InvariantCulture for every numeric parse/format and ordinal string comparisons. This is a correctness requirement of the format.
  • Invariant-culture enforcement: the rule above is enforced by a repo-root .editorconfig that raises the .NET globalization analyzers (CA1305, CA1304, CA1311, CA1310, CA1309) to build errors, so any culture-sensitive format/parse/casing/comparison must state an explicit invariant/ordinal choice. Caveat: these analyzers do not catch culture-sensitive string interpolation of numbers ($"{value}"), and they enforce explicitness rather than the specific invariant/ordinal selection. So numbers must never be raw-interpolated: format them explicitly with CultureInfo.InvariantCulture. A culture-swap test (run under a non-invariant culture such as fr-FR, asserting the exact expected string) is required wherever the formatting is actually culture-sensitive — floating-point/decimal values, or any grouped numeric format — which first arises with float serialization in M4. A plain integer in the default format is culture-robust (no digit grouping in any culture), so its explicit invariant formatting is defensive only and needs no required culture-swap test.
  • Constants: the FITS size constants live on FITSFile as public const (BlockSize = 2880, CardSize = 80, KeywordLength = 8, FixedValueFieldWidth = 20, MaxDataSize = 1L << 53) to preserve the FITSFile.BlockSize API. FITSFile.cs is created early holding only these constants and completed in the later file-layer milestones.
  • Async: the library is synchronous; the Swift tests are async throws only as a Swift Testing idiom. The C# port and its tests are synchronous ([Fact] methods, no async).

Porting hazards (read before implementing)

  • Float serialization must keep a decimal point or exponent. Swift renders float(42.0) as "42.0" (via Double.description), which re-parses as a float. A naive double.ToString(InvariantCulture) yields "42", which the value classifier matches as an integer first — silently changing the type on round-trip. The C# renderer must guarantee the output contains ., E, or e (append .0 when absent) so float values round-trip as floats. This is a case where the “42.0 vs 42” difference the source doc calls acceptable is not acceptable, because it is behavior tested by the original project.
  • padding(toLength:withPad:startingAt:) both pads and truncates. Swift's method truncates when the string is longer than the target; C#'s PadRight only pads. A shared helper must replicate pad-or-truncate semantics (used for keyword fields, the END marker, and card padding).
  • CharacterSet has no .NET equivalent. The Swift fitsPadding, fitsPaddingWithNul, and fitsKeyword sets become an internal FITS-charset helper (membership predicates over char). The public trimming extensions take a set/predicate rather than a CharacterSet.
  • Property → section back-reference is a plain reference, not weak. Swift uses weak var section to avoid a reference-counting cycle; the .NET GC collects cycles, so a plain nullable reference (FITSSection?) is correct and simpler. The dirty-tracking hook (value/comment setters calling section?.MarkNeedsSerialization()) is preserved.
  • Lazy cached flags. FITSBlock's lazy var structural flags become lazily-computed cached properties (nullable backing fields). The Swift “not thread-safe” caveat is inherited and documented, not fixed.
  • Regex. The integer / floating-point grammar patterns port verbatim to System.Text.RegularExpressions (prefer [GeneratedRegex]), compiled once and shared, using the same patterns.
  • File reading. Data(contentsOf:.mappedIfSafe) maps to File.ReadAllBytes (simple and safe) — or MemoryMappedFile if the mmap path is wanted to fully exploit the ReadOnlyMemory<byte> zero-copy design. The Swift SIGBUS note is dropped. A URL becomes a file path; failure is classified missing/directory → InvalidFileURL, otherwise CannotReadFile.
  • ASCII length model. A FITS record is 80 bytes; the parser guards that records are ASCII so Swift Character counts and C# string.Length (UTF-16 units) both coincide with the byte length. Keep the ASCII guard so all length/slicing math stays valid.
  • OptionSet → [Flags] enum. The two option types become [Flags] enums; Strict/Lenient are combined named values. options.contains(.x) becomes options.HasFlag(...) (or a bitwise test).
  • Zero-based slicing. Because ReadOnlyMemory<byte>.Slice re-bases to index 0, the Swift gotcha of slices carrying a non-zero startIndex disappears — but the Chunked(by:) and block-scanning logic must be written against offsets/lengths, not absolute indices.

Test framework mapping

Swift Testing → xUnit v3. See the appendix for the full table. Highlights: @Test → [Fact]; #expect(x) → Assert.True / Assert.Equal; #require(x) → unwrap-or-fail (Assert.NotNull then use); #expect(throws: FITSError.self){ } → Assert.Throws<FITSException>(() => ...); #expect(throws: Never.self){ } → call directly (the test fails if it throws). @testable import is covered by the existing InternalsVisibleTo.

Appendix

Dependency order

Each milestone depends only on earlier ones, so the build stays green at every step:

constants (M1)
  → FITSException, options (M2)
    → charset/string/memory helpers, test utils (M3)
      → FITSValue (M4)
        → FITSBlock (M5)
        → FITSProperty parse (M6) → FITSProperty serialize (M7)
          → FITSSection (M8)   [needs property serialize]
            → FITSFile parse (M9) → FITSFile build/write (M10)
              → integration (M11) → finalization (M12)

Splitting FITSProperty (M6/M7) and FITSFile (M1/M9/M10) by concern is safe because a later-milestone method is never referenced by earlier code: FITSSection is the first caller of property serialization, and nothing calls the file build/write path before M10.
